# Полное ревью проекта от 23.07.2026

## Цель

Повторно проверить проект после последнего крупного коммита, восстановить замысел предыдущего ревью, завершить незаконченные улучшения и исправить найденные ошибки без изменения:

- модели, таблицы и админ-класса «Временные учетные данные» (`TemporaryCredential`);
- сценария смены пароля без ввода старого пароля.

## Что удалось восстановить из истории Git

Предыдущее ревью шло по следующим направлениям:

1. **Жизненный цикл временных учетных данных и регистрация**
   - привязка временных данных к пользователю и заявке;
   - безопасная обработка регистрации;
   - ограничение частоты публичных заявок;
   - очистка временного доступа после установки нового пароля.

2. **Архив учебных лет**
   - только один активный учебный год;
   - архивные годы доступны только для чтения;
   - отдельные годовые зачисления учеников;
   - снимки ФИО, групп, предметов и преподавателей для сохранения истории;
   - проверка тестовых данных на принадлежность активному году;
   - инструкция по переходу на новый учебный год.

3. **Эксплуатация проекта**
   - CI/CD;
   - Docker-запуск;
   - резервные копии PostgreSQL;
   - административные инструменты данных.

Последний крупный коммит реализовал большую часть этих пунктов, но часть сценариев жизненного цикла учебного года, production-конфигурации и обслуживания оставалась неполной.

## Найденные и исправленные проблемы

### 1. Переключение активного учебного года

**Проблема:** логика выбора последнего года была распределена по нескольким местам. При редактировании дат активного года или возврате к предыдущему году архивные снимки могли не финализироваться, а текущие группы учеников — не восстанавливаться из годовых зачислений.

**Исправлено:**

- активация централизована в `AcademicYear`;
- добавлена транзакционная блокировка переключения;
- для PostgreSQL используется advisory lock, дополнительно блокируются строки учебных лет;
- при смене активного года финализируются снимки уходящего года;
- группа и активность ученика синхронизируются с зачислением выбранного года;
- при удалении пустого последнего года восстанавливается предыдущий;
- добавлены регрессионные тесты этих сценариев.

### 2. Конфигурация Django 5.2 и WhiteNoise

**Проблема:** проект использовал удалённую настройку `STATICFILES_STORAGE`. При текущей ветке Django 5.2 это не гарантировало корректную production-конфигурацию WhiteNoise.

**Исправлено:**

- настройка переведена на `STORAGES`;
- для статики задан `CompressedManifestStaticFilesStorage`;
- CI теперь выполняет production security check и `collectstatic`.

### 3. Права непривилегированного пользователя Docker

**Проблема:** контейнер запускается от пользователя `app`, но каталог `/app` мог оставаться недоступным для записи. На чистом образе это могло приводить к падению `collectstatic` в entrypoint.

**Исправлено:**

- создаётся `/app/staticfiles`;
- рабочий каталог перед запуском передаётся пользователю `app`;
- Docker job в CI теперь загружает собранный образ и проверяет права, `manage.py check` и `collectstatic` внутри реального контейнера.

### 4. Production-переменные базы данных

**Проблема:** в примере production-env значения `DB_NAME`/`DB_USER` и `POSTGRES_DB`/`POSTGRES_USER` могли расходиться. В таком случае PostgreSQL создавал одну базу, а Django пытался подключиться к другой.

**Исправлено:**

- пары переменных согласованы;
- удалены неиспользуемые SSH/GHCR-поля из env-примеров;
- локально собираемый образ задан как `cadet-journal:web`;
- для пароля инструментов данных используется единое имя `DATA_TOOLS_PASSWORD`.

### 5. Создание `.env.prod` при CD

**Проблема:** при прерывании workflow временный файл с секретами мог остаться рядом с `.env.prod`.

**Исправлено:**

- добавлена очистка временного файла через `trap`;
- рабочий env по-прежнему заменяется атомарно;
- обязательные значения проверяются до деплоя;
- опасные инструменты данных принудительно отключаются в production.

### 6. Резервное копирование и восстановление PostgreSQL

**Проблемы:** архив не проверялся перед сохранением/восстановлением, а ошибка в середине восстановления могла оставить базу частично изменённой.

**Исправлено:**

- резервная копия сначала создаётся во временный файл;
- архив проверяется через `pg_restore --list`;
- готовый файл перемещается атомарно;
- восстановление использует `--exit-on-error` и одну транзакцию;
- добавлена очистка объектов через `--clean --if-exists`;
- параметры и сроки хранения валидируются;
- документация восстановления обновлена.

### 7. Публичная регистрация

**Проблема:** обработчик JSON мог принять массив вместо объекта или вернуть ошибку сервера на некорректной UTF-8-последовательности.

**Исправлено:**

- JSON разбирается только при соответствующем `Content-Type`;
- перехватываются ошибки декодирования и JSON;
- корневое значение обязано быть объектом;
- добавлены тесты массива JSON и некорректной UTF-8.

### 8. Экспорт данных

**Проблема:** read-only XLSX-обработчики не ограничивали HTTP-метод и могли отвечать на POST.

**Исправлено:**

- три XLSX-экспорта принимают только GET;
- добавлены тесты ответа `405 Method Not Allowed`;
- доступ остаётся только у суперпользователя;
- обработка Excel-формул остаётся защищённой.

### 9. Зависимости и проверки качества

**Исправлено:**

- production-зависимости обновлены и закреплены;
- удалён неиспользуемый Gunicorn, поскольку контейнер запускается через Uvicorn;
- создан `requirements-dev.txt` с закреплённым Ruff;
- CI выполняет `pip check`;
- удалены неиспользуемые импорты;
- `reverse()` перехватывает только `NoReverseMatch`, а не любые исключения;
- compatibility-тесты перехватывают конкретный `CommandError`.

### 10. Скрипты обслуживания и документация

**Исправлено:**

- `seed_all.sh` действительно выполняет миграции перед заполнением и работает как на хосте, так и внутри контейнера;
- `exp_tc.sh` принимает путь результата, создаёт каталог, очищает временный файл контейнера и больше не запускает `nano`;
- `run-local.sh` поддерживает Docker через `sudo -n`;
- `bootstrap-server.sh` не зависит от жёстко заданных имён контейнеров и использует общий `run-prod.sh`;
- README приведён в соответствие с фактическими правами доступа и командами;
- инструкция CD переписана под реальную схему сборки на сервере без GHCR.

## Что нельзя было динамически проверить в текущей среде

В рабочей среде отсутствовали Django и Docker, а установка Python-зависимостей была недоступна из-за отсутствия сетевого DNS-доступа. Поэтому локально не были выполнены:

- полный `python manage.py test`;
- `manage.py makemigrations --check`;
- реальная сборка и запуск Docker-образа;
- восстановление в настоящий PostgreSQL.

Для этих проверок CI усилен: он устанавливает закреплённые зависимости, проверяет миграции, запускает system/security checks, все Django-тесты, собирает статику, собирает Docker-образ и поднимает production-подобный контейнер с реальным HTTP healthcheck. На момент формирования архива эти новые коммиты ещё не отправлены в GitHub, поэтому обновлённый CI для них пока не выполнялся.

## Рекомендация перед production-деплоем

Первый запуск обновлённого CI должен завершиться успешно. После этого на staging-копии базы следует один раз проверить полный цикл: создать backup, восстановить его и сверить ключевые количества записей. Это единственная значимая проверка, которую нельзя достоверно заменить статическим анализом.


## Дополнительная проверка после выдачи архива

Повторная проверка выявила и закрыла ещё три риска:

- ученики без зачисления в новый активный год теперь деактивируются и теряют текущую группу, а при возврате года состояние восстанавливается из зачисления;
- CD запускается только после успешного CI для push в `main` и разворачивает точный проверенный SHA;
- production-порт привязан к `127.0.0.1`, а обязательная схема HTTPS reverse proxy явно задокументирована;
- healthcheck теперь передаёт допустимые `Host` и `X-Forwarded-Proto`, поэтому не ломается из-за production HTTPS-настроек;
- rate limit регистрации получает адрес клиента с доверенной правой стороны `X-Forwarded-For`, а production включает один доверенный proxy-hop;
- секреты CD записываются в Compose env-файл без интерполяции `$` и без риска частично отменённого параллельного деплоя;
- production-запуск и локальный Docker-запуск ждут готовности healthcheck, а CD сбрасывает любые серверные правки до переключения на проверенный SHA;
- `seed_all.sh` получил исполняемый бит и соответствует командам из README.
