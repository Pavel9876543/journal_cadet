## Результат по требованиям

### 1. Ученики в карточке группы

Во вкладке учеников группы теперь используется поле выбора ученика по ФИО.

Поддержано:

- выбор существующего ученика;
- создание новой карточки ученика из related-окна;
- изменение выбранного ученика;
- удаление карточки ученика через стандартное related-окно Django Admin;
- переход по кликабельному ФИО в карточку ученика;
- автоматическое заполнение поля «Город / Церковь» из выбранной карточки;
- увеличенное окно для экранов шириной от 992 px;
- увеличенная Bootstrap/Jazzmin-модаль и обычное popup-окно Django Admin.

Для архивного года строки состава группы показываются только для чтения по сохраненным снимкам.

### 2. Поле «Город / Церковь»

Поле расширено в:

- карточке ученика;
- заявке на курсы;
- составе группы;
- публичной форме регистрации;
- табличных inline-формах админки.

На мобильных устройствах ширина остается адаптивной, на ПК поле получает увеличенную минимальную и максимальную ширину.

### 3. Сброс текущей группы при создании нового года

При активации нового учебного года:

- у учеников, не зачисленных в новый год, текущая `group` становится `None`;
- `is_active` становится `False`;
- зачисления, оценки, итоги, предметы и снимки прошлого года сохраняются;
- при переключении обратно на год текущее состояние восстанавливается из `StudentEnrollment`.

### 4. Редактирование архивного года

В архивном году:

- общие поля карточек ученика и преподавателя остаются редактируемыми;
- контакты, анкета и связанный аккаунт остаются редактируемыми;
- годовые назначения, зачисления, предметы, оценки, итоги и другие учебные данные доступны только для просмотра;
- добавление новых записей отключено;
- удаление записей отключено;
- массовые действия отключены;
- серверные методы сохранения и удаления дополнительно блокируют поддельные POST-запросы;
- таблица временных данных доступна только для просмотра;
- глобальные справочники также нельзя менять, пока в админке выбран архивный год.

### 5. Ошибка архивной заявки `KeyError: 'last_name'`

Форма заявки больше не обращается без проверки к полям, исключенным админкой в режиме просмотра. Архивная заявка открывается без `KeyError`, даже когда все поля заявки переведены в readonly.

Также устранено повторное выполнение `super().save()`, из-за которого заявка могла сохраняться дважды.

### 6. Переключение учебных годов

Добавлен выбор учебного года:

- на главной странице админки;
- в списках моделей;
- в формах изменения;
- в личном журнале ученика;
- в личном журнале преподавателя;
- в журнале администратора.

Выбор хранится в сессии. По умолчанию выбирается активный год, а при его отсутствии — последний доступный.

Все годовые таблицы фильтруются по выбранному году. Главная панель, статистика и Excel-выгрузка временных данных также учитывают выбранный год.

### 7. Общие пользователи и годовая история

Профиль ученика или преподавателя и его учетная запись являются общими сущностями. Годовое участие хранится отдельно:

- ученик — `StudentEnrollment`;
- преподаватель — `TeacherEnrollment`;
- групповые назначения — через группу выбранного года;
- индивидуальные предметы, оценки и итоги — с явным учебным годом.

Повторная заявка идентифицирует ученика по нормализованному ФИО и дате рождения:

- пробелы нормализуются;
- регистр не учитывается;
- `ё` и `е` считаются эквивалентными;
- существующие `Student` и `User` переиспользуются;
- пароль не меняется;
- новый временный пароль не создается;
- создается только участие в новом учебном году.

Пользователь видит в личном кабинете только годы своего фактического участия. Ученик или преподаватель, появившийся впервые в новом году, не появляется в старом году.

Исторические ФИО, названия предметов и преподавателей выводятся из снимков, поэтому последующее переименование общей карточки не искажает архив.

### 8. Регистрация только в активный год

Публичная форма создает заявку только для активного учебного года. Если активного года нет, регистрация не продолжается и возвращает понятное сообщение.

Архивная заявка не может быть изменена, подтверждена, отклонена или удалена.

### 9. Телефон в разных годах

Уникальность телефона действует по паре:

```text
(academic_year, student_phone)
```

Одинаковый нормализованный телефон разрешен в разных учебных годах, но запрещен дважды в одном году.

При гонке двух одновременных заявок защиту обеспечивает уникальное ограничение PostgreSQL. Второй запрос получает контролируемый ответ `409`, а не `500`.

Для повторных заявок одного человека с разными телефонами добавлен конкурентный PostgreSQL-тест: две заявки могут существовать, но должны использовать одного ученика, один аккаунт и один первоначальный временный пароль.

### 10. Даты курсов

Поля дат начала и окончания удалены из `CourseRegistrationSettings`. Регистрация, расчет возраста и отображение периода используют `starts_on` и `ends_on` активного `AcademicYear`.

Исторические функции значений по умолчанию оставлены только потому, что на них ссылаются старые миграции и они нужны для развертывания базы с нуля.

### 11. Глобальные настройки регистрации и восстановления

Проверено:

- `CourseRegistrationSettings` остается глобальной singleton-настройкой;
- минимальный возраст и Telegram-ссылка не принадлежат учебному году;
- `PasswordRecoveryContact` остается глобальной таблицей;
- при выборе архивного года эти настройки не копируются и не фильтруются как учебные записи.

### 12. Неизменность неактивного учебного года

Существующий неактивный `AcademicYear` нельзя изменить или удалить. Модель проверяет это независимо от интерфейса админки. Добавление более нового года автоматически делает его активным и архивирует предыдущий.

## Дополнительные исправления ревью

- введено годовое участие преподавателей;
- исправлено начальное заполнение активностей преподавателей миграцией данных;
- отклонение одной из нескольких подтвержденных заявок не удаляет общий аккаунт, зачисление или единственный временный пароль;
- временный пароль в таблице показывается только в году создания аккаунта, а не во всех годах его дальнейшего обучения;
- Excel-выгрузки временных данных учитывают выбранный год;
- устранены одинаковые имена тестовых методов, из-за которых Python незаметно перекрывал один тест другим;
- счетчик предметов на главной админки теперь считает только фактически назначенные предметы выбранного года;
- подсказки отклонения заявки приведены в соответствие с безопасным сохранением общих данных.

## Пароли и временные данные

Сохранены прежние ограничения проекта:

- схема `TemporaryCredential` не изменена;
- временный пароль создается только вместе с новым аккаунтом;
- редактирование ФИО, контактов, группы, роли, активности или аккаунта не меняет хэш пароля;
- повторная заявка существующего человека не меняет пароль;
- форма смены пароля наследуется от `SetPasswordForm` и не требует старый пароль.

## Ограничение динамической проверки

В исполнительной среде отсутствуют Docker/Podman и пакет Django. Доступ к реестру Python-пакетов также недоступен. Поэтому здесь невозможно честно выполнить реальный запуск контейнеров, миграции PostgreSQL и `manage.py test`.

Поскольку пользователь потребовал `[skip ci]`, GitHub Actions для новых коммитов также намеренно не запускается.

Перед использованием на рабочей машине необходимо выполнить:

```bash
./scripts/run-local.sh

docker compose \
  --env-file .env.dev \
  -f docker-compose.yml \
  -f docker-compose.dev.yml \
  exec -T web python manage.py check

docker compose \
  --env-file .env.dev \
  -f docker-compose.yml \
  -f docker-compose.dev.yml \
  exec -T web python manage.py makemigrations --check --dry-run

docker compose \
  --env-file .env.dev \
  -f docker-compose.yml \
  -f docker-compose.dev.yml \
  exec -T web python manage.py test -v 2
```

После тестов следует отдельно проверить в браузере:

1. создание нового учебного года и сброс текущих групп;
2. открытие архивной заявки;
3. related-окно ученика в группе на ПК;
4. переключение года для администратора, ученика и преподавателя;
5. повторную заявку существующего ученика в новом году.
