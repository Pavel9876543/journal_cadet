# Повторное ревью проекта — исправление контейнера, учетных данных и конкурентной регистрации

Дата: 23.07.2026

## Исходная проблема

Контейнер `web` завершался во время `collectstatic` с ошибкой:

```text
PermissionError: [Errno 1] Operation not permitted: '/app/staticfiles/...'
```

Development Compose примонтирует каталог проекта с Windows в `/app`. Django собирал статику в `/app/staticfiles` и пытался выполнить `chmod` на файлах Windows bind mount. Такая операция не поддерживается используемой файловой системой Docker Desktop. Изменение владельца `/app` внутри образа не решает проблему, потому что bind mount скрывает содержимое и права каталога из образа.

## Этап 1. Исправление запуска контейнера

Исправлено:

- `STATIC_ROOT` перенесен в `/var/lib/cadet-journal/staticfiles`, за пределы `/app`;
- каталог создается и передается непривилегированному пользователю `app` при сборке образа;
- значение закреплено одновременно в Dockerfile, базовом Compose и Django settings;
- entrypoint проверяет доступность каталога для записи и выводит понятную ошибку;
- `collectstatic` выполняется с `--clear`;
- CI проверяет сборку статики внутри образа и при read-only bind mount исходников в `/app`.

После получения новой версии старый образ нужно пересобрать. Рекомендуемая команда:

```bash
./scripts/run-local.sh
```

## Этап 2. Создание пользователей и таблица временных данных

Проверены и унифицированы поддерживаемые пути создания аккаунтов.

### Новый администратор

Запись `TemporaryCredential` создается при:

- создании пользователя через стандартный раздел пользователей Django Admin;
- запуске `python manage.py createsuperuser`;
- первом запуске `python manage.py ensure_superuser`, если пользователя еще нет.

`createsuperuser` выполняется транзакционно: пользователь, группа администратора и временные данные либо создаются вместе, либо вся операция откатывается.

### Новый преподаватель

Запись создается при:

- ручном создании преподавателя без выбранного аккаунта в админке;
- запуске `python manage.py create_teacher_accounts` для преподавателя без аккаунта;
- создании нового демонстрационного преподавателя через `seed_data`.

### Новый ученик

Запись создается при:

- ручном создании ученика без выбранного аккаунта в админке;
- запуске `python manage.py create_student_accounts` для ученика без аккаунта;
- подтверждении новой заявки на курсы;
- создании нового демонстрационного ученика через `seed_data`.

Если к профилю привязан уже существующий пользователь, новый временный пароль не назначается. Это намеренное поведение: открытый пароль существующего аккаунта неизвестен, а его автоматический сброс нарушил бы доступ пользователя.

Низкоуровневый вызов `User.objects.create_user(...)` напрямую из произвольного shell-кода не создает строку временных данных автоматически, поскольку Django сохраняет только хэш и после создания не располагает открытым паролем. Для терминала необходимо использовать management-команды проекта.

## Этап 3. Неизменность паролей при редактировании данных

Зафиксировано правило: временный пароль может быть передан в сервис временных данных только при явном создании нового аккаунта.

При изменении существующего пользователя не меняются:

- хэш пароля Django;
- значение `temporary_password` в существующей строке;
- наличие строки временных данных, если ее раньше не было.

Проверены изменения:

- логина и email пользователя;
- ФИО и контактов преподавателя;
- ФИО, телефона, группы и других данных ученика;
- повторное подтверждение или редактирование заявки;
- повторные запуски `ensure_superuser`, `create_teacher_accounts` и `create_student_accounts`.

Удалены режимы, которые могли менять пароль существующего аккаунта:

- `--reset-existing` у команд учеников и преподавателей;
- `DJANGO_SUPERUSER_ROTATE_PASSWORD`;
- аварийное создание нового пароля при отсутствии строки временных данных.

Форма обычной смены пароля не изменена: она наследуется от `SetPasswordForm`, не требует старый пароль и после успешной смены удаляет выданные временные данные.

## Этап 4. Одновременные заявки с одинаковым телефоном

На уровне PostgreSQL уже действует ограничение:

```text
unique_course_app_phone_per_year
```

Оно запрещает две заявки с одинаковым нормализованным номером телефона в одном учебном году, независимо от того, насколько одновременно пришли запросы.

Дополнительно исправлена обработка гонки:

- обычный ранний дубль определяется проверкой модели;
- если оба запроса успели пройти раннюю проверку, PostgreSQL разрешает только одну вставку;
- второй запрос распознает имя ограничения и получает контролируемый ответ `409`, а не ошибку `500`;
- посторонние `ValidationError` и `IntegrityError` не маскируются;
- добавлен PostgreSQL `TransactionTestCase` с двумя потоками и синхронизацией перед вставкой.

## Асинхронность базы данных

HTTP-обработчики проекта объявлены как `async` и работают через ASGI/Uvicorn. Сложная ORM-логика и транзакции остаются синхронными и передаются через `sync_to_async` в рабочие потоки. Для PostgreSQL разные запросы могут выполняться параллельно на отдельных соединениях.

Это не нативный async-драйвер ORM, но это корректная для текущей архитектуры схема: event loop не блокируется, а транзакционные методы моделей остаются атомарными. Защиту от дублей обеспечивает не порядок выполнения потоков, а ограничение PostgreSQL.

## Ограничение проверки

В текущей исполнительной среде отсутствуют Docker CLI/daemon и установленные зависимости Django. Сетевой доступ к реестру Python-пакетов также недоступен. Поэтому здесь невозможно честно выполнить:

- реальную сборку нового Docker-образа;
- `docker compose up` с PostgreSQL;
- полный `python manage.py test`;
- PostgreSQL-тест с двумя реальными соединениями.

Эти проверки добавлены в проект и CI, но из-за требуемого `[skip ci]` новые коммиты автоматически их не запустят. Для окончательной динамической проверки на машине с Docker следует выполнить:

```bash
./scripts/run-local.sh
docker compose --env-file .env.dev -f docker-compose.yml -f docker-compose.dev.yml exec -T web python manage.py check
docker compose --env-file .env.dev -f docker-compose.yml -f docker-compose.dev.yml exec -T web python manage.py makemigrations --check --dry-run
docker compose --env-file .env.dev -f docker-compose.yml -f docker-compose.dev.yml exec -T web python manage.py test -v 2
```

После `./scripts/run-local.sh` в логах не должно быть путей `/app/staticfiles`; сборка должна идти в `/var/lib/cadet-journal/staticfiles`.
